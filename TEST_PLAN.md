# [테스트 계획서] LangGraph 기반 지능형 여행 플래너 에이전트

## 1. 개요
* **목적**: 사용자의 여행 의도 파악부터 장소 검색, 일정 생성까지 이어지는 멀티 에이전트 워크플로우의 안정성 및 데이터 정합성 검증.
* **대상 시스템**: LangGraph 기반 LLM 에이전트 및 RAG(Self-Querying + Rerank) 검색 파이프라인.
* **핵심 지표**: Intent 라우팅 정확도, 목적지 변경 시 상태 초기화 여부, 검색 결과의 시맨틱 유사도.

## 3. 테스트 환경 및 도구
* **모델**: GPT-4o-mini, GPT-4.1, GPT-4.1-mini
* **프레임워크**: LangGraph, Streamlit
* **데이터베이스**: ChromaDB (Vector Store)
* **평가 도구**: RAGAS (RAG 평가 프레임워크) 또는 정성적 사용자 피드백

---

## 4. 결함 관리 및 우선순위
* **P0 (Critical)**: 목적지 전환 시 이전 도시의 데이터가 노출되는 경우 (데이터 오염)
* **P1 (Major)**: 일정 생성 시 시간대가 겹치거나 장소 간 거리가 비정상적으로 먼 경우
* **P2 (Minor)**: 챗봇의 말투가 일관되지 않거나 UI 레이아웃이 깨지는 경우

---

## 5. 승인 기준
1. 모든 P0, P1 결함이 해결되어야 함.
2. 사용자 의도에 따른 라우팅 성공률 90% 이상 달성.
3. 목적지 변경 시 데이터 초기화 및 교체 로직의 정상 작동 확인.

### 6.1. 상세 결함 조치 리포트

#### 6.1.1 [Critical] 목적지 변경 시 데이터 오염(Data Pollution) 이슈
* **결함 내용**: '경주' 일정 생성 후 '부산'으로 변경 시 제목만 부산이고 내용은 경주 정보가 출력되는 현상.
* **원인 분석**: `select_places_node`에서 기존 데이터와 신규 목적지 간의 정합성 검증 단계 미흡.
* **조치 내용**: 
    - `select_places_node` 내에 목적지 불일치 감지 로직 추가.
    - 목적지 변경 시 관련 리스트(`itinerary`, `selected_places` 등)를 강제 초기화하도록 `Reducer` 로직 보완.
* **테스트 결과**: 목적지 전환 시 100% 확률로 데이터 초기화 및 신규 데이터 매핑 확인 (**Pass**).

#### 6.1.2 [Major] 대화 문맥 유지 및 State 유실 이슈
* **결함 내용**: 목적지 확정 후 "내일 갈 거야"와 같이 날짜를 추가 입력하면 이전 목적지 정보를 망각함.
* **조치 내용**: 
    - **Node 보완**: `extract_trip_requirements_node`에서 신규 입력에 목적지가 없을 시 기존 값을 명시적으로 유지하는 `elif` 구문 추가.
    - **State 선언**: `destination` 타입을 `Annotated`로 변경하고 `keep_and_update` 함수를 적용하여 업데이트 기능 명시.
    - **Invoke 수정**: `chat_logic.py`에서 `graph.invoke` 시 메시지뿐만 아니라 세션에 저장된 기존 상태 값을 함께 전달하도록 수정.
* **테스트 결과**: 추가 제약 사항 입력 시에도 목적지 정보가 정상 유지됨 (**Pass**).

#### 6.1.3 [Major] 상세 지명 미인식으로 인한 목적지 누락
* **결함 내용**: "해운대 가고 싶어"라고 입력할 경우, '부산'이라는 키워드가 없어 목적지 추출에 실패함.
* **원인 분석**: 텍스트 매칭 기반의 단순 추출 로직 한계.
* **조치 내용**: `sub_locations` 매핑 사전을 도입하여 하위 지역명 입력 시 상위 행정구역으로 자동 치환하는 전처리 로직 구현.
* **테스트 결과**: 주요 관광지/동네 이름 입력 시 상위 도시가 정상적으로 State에 저장됨 (**Pass**).

#### 6.1.4 [Normal] 검색 결과의 시맨틱 매칭 품질 저하
* **결함 내용**: 단순 벡터 유사도 검색 결과가 사용자의 세부 요구사항(예: 조용한, 부모님과 함께)을 충분히 반영하지 못함.
* **원인 분석**: 임베딩 유사도만으로는 복합적인 뉘앙스 파악에 한계가 있음.
* **조치 내용**: 
    - `SelfQueryRetriever`를 통한 메타데이터 1차 필터링 수행.
    - `Reranker` 노드를 추가하여 상위 후보군을 사용자 쿼리 관점에서 재정렬.
* **테스트 결과**: 정성 평가 결과 추천 장소의 의도 부합도가 기존 대비 약 30% 향상됨 (**Pass**).

#### 6.1.5 [Normal] 검색 결과 내 동일 장소 중복 노출 이슈
* **결함 내용**: 벡터 DB 검색 시 동일한 장소에 대한 여러 리뷰가 개별 검색 결과로 취급되어 UI에 중복된 장소가 나열됨.
* **원인 분석**: 검색 엔진(Retriever)이 장소 단위가 아닌 텍스트 청크(리뷰 단위)로 결과를 반환함에 따라 발생하는 현상.
* **조치 내용**: 
    - `db_retrieval` 과정에서 `seen_place_ids` (Set 자료구조)를 활용한 필터링 로직 구현 및 `trip_nodes`의 `select_places` 함수에 `seen_names`로 동일한 지명이나 다른 카테고리 (예시: 해운대 - default, 해운대 - beach)로 인해 중복 검색된 장소 구분.
    - 검색 결과 리스트를 순회하며 이미 처리된 장소 ID는 스킵하도록 설계하여 유니크한 장소만 UI로 전달.
* **테스트 결과**: 다수의 리뷰가 존재하는 장소 검색 시에도 UI에는 단일 장소 정보만 정상 노출됨 (**Pass**).

#### 6.1.6 [Normal] 상세 조건(예: 서핑) 반영 미흡 및 키워드 희석 이슈
* **결함 내용**: 사용자가 '서핑' 같은 구체적인 활동을 요구했으나, 일정에는 '산책' 등 일반적인 액티비티만 포함됨.
* **원인 분석**: 상세 키워드가 상위 범주(액티비티)로 정규화되면서 검색 우선순위에서 밀려나는 현상 발생.
* **조치 내용**: 
    - **추출 강화**: 스타일 정규화 시 사용자 입력 키워드를 원형 보존하도록 프롬프트 수정 및 규칙 기반 추출 보완.
    - **검색 최적화**: 검색 쿼리 생성 시 '서핑' 같은 개성적 키워드를 '액티비티'보다 앞선 순위로 재정렬.
    - **검증 고도화**: `validate_node`의 프롬프트에 '사용자가 요청한 핵심 활동을 반드시 하나 이상 포함'하도록 지침 강화.
* **테스트 결과**: 상세 활동 키워드가 검색 및 일정 생성 전 과정에 누락 없이 반영됨을 확인 (**Pass**).

### 6.2. 종합 의견
초기 개발 단계에서 발생했던 데이터 정합성, 문맥 유지, 중복 데이터 필터링 문제를 모두 해결하였습니다. 특히 멀티 에이전트 환경에서 State 관리를 최적화함으로써 사용자 인터랙션에 따른 유연한 대응이 가능해졌습니다.
또한, Reranker와 같은 복합 파이프라인을 포함하고 있음에도 불구하고 챗봇의 전체 반응 시간을 약 20초 이내로 제어하여 서비스 안정성과 사용자 경험을 동시에 확보하였습니다. 본 테스트 결과에 따라 프로젝트가 최종 승인 기준을 모두 충족하였음을 확인하며, 이후 실사용자 피드백을 바탕으로 검색 쿼리를 지속적으로 고도화할 예정입니다.
